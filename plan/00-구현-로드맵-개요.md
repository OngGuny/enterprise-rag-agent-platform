## 추천 구현 순서

### Phase 1: 기반 (01 + 02 + 03)
- 프로젝트 골격과 설정
- DB 연결, ORM 모델, 리포지토리
- 예외 처리, 미들웨어, DI

**완료 기준:** `docker compose up`으로 PostgreSQL, Redis, Milvus가 뜨고, `/api/v1/health/ready`가 모든 서비스 OK 반환

### Phase 2: RAG 코어 (04)
- 텍스트 추출기 (PDF 우선)
- Parent-Child 청킹
- Ollama 임베딩 (bge-m3)
- Milvus Hybrid Search + RRF
- Re-ranking

**완료 기준:** PDF 업로드 → 청킹 → 임베딩 → Milvus 저장 → 질문으로 검색하여 관련 문서 반환

### Phase 3: LLM + 채팅 (05 + 07 일부)
- LLM 클라이언트 (Ollama, OpenAI)
- 모델 라우팅
- SSE 스트리밍
- Chat API
- 시맨틱 캐시

**완료 기준:** `/api/v1/chat`에 질문 → RAG 검색 + LLM 답변이 SSE로 스트리밍

### Phase 4: 에이전트 (06 + 07 일부)
- AgentState, LangGraph 그래프
- 노드 구현 (analyze, rag, web, sql, generate, guardrails)
- Tool Calling
- Agent API

**완료 기준:** `/api/v1/agents/execute`로 복합 질문 → 멀티스텝 에이전트 실행 → 답변

### Phase 5: 인프라 마무리 (08)
- Docker Compose 통합
- Nginx SSE 프록시
- Prometheus + Grafana
- 테스트

**완료 기준:** 전체 서비스가 Docker Compose로 구동, Grafana에서 API 메트릭 확인

## 설계 원칙 (프로덕션 레벨)

1. **인터페이스 우선**: 모든 핵심 컴포넌트에 Base 클래스(ABC) 정의 → 구현 교체 가능
2. **비동기 기본**: `async/await` 전면 적용, I/O-bound 작업은 모두 비동기
3. **실패 격리**: 서킷 브레이커로 외부 서비스 장애 전파 차단
4. **관측 가능성**: structlog JSON 로깅 + Prometheus 메트릭 + request_id 추적
5. **점진적 열화**: 서비스 일부 장애 시 degraded 모드 (예: 캐시 다운 → 캐시 우회)
6. **테스트 가능성**: DI로 모든 외부 의존성 교체 가능, 리포지토리 패턴으로 DB 추상화
