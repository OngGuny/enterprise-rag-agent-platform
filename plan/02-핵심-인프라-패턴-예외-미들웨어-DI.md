## 2.1 커스텀 예외 체계 (core/exceptions.py)

### 예외 계층 설계

```
AppException (base)
├── ValidationError          # 400 - 입력 검증 실패
├── AuthenticationError      # 401 - 인증 실패
├── AuthorizationError       # 403 - 권한 부족
├── NotFoundError            # 404 - 리소스 없음
├── ConflictError            # 409 - 상태 충돌
├── RateLimitExceededError   # 429 - 요청 한도 초과
├── ExternalServiceError     # 502 - 외부 서비스 장애
│   ├── LLMServiceError
│   ├── MilvusServiceError
│   └── EmbeddingServiceError
└── InternalError            # 500 - 내부 오류
```

### 구현 원칙

- 모든 커스텀 예외는 `AppException`을 상속
- 각 예외에 `error_code` (문자열), `status_code` (HTTP), `detail` (사용자 메시지), `context` (디버그 정보) 포함
- context는 로그에만 남기고, 클라이언트 응답에는 포함하지 않음 (정보 노출 방지)

### FastAPI Exception Handler

```python
@app.exception_handler(AppException)
async def app_exception_handler(request: Request, exc: AppException):
    logger.error(
        "app_error",
        error_code=exc.error_code,
        status_code=exc.status_code,
        detail=exc.detail,
        context=exc.context,
        path=request.url.path,
    )
    return JSONResponse(
        status_code=exc.status_code,
        content={
            "error": {
                "code": exc.error_code,
                "message": exc.detail,
            }
        },
    )
```

### 프로덕션 포인트

- 예상치 못한 예외(Exception)도 핸들러로 잡아서 500 + 구조화된 JSON 반환
- Pydantic `ValidationError`는 별도 핸들러로 422 → 필드별 에러 메시지 포맷팅
- 스택 트레이스는 로그에만 남기고, 응답에는 generic 메시지만

## 2.2 요청 로깅 미들웨어

### RequestLoggingMiddleware

```python
class RequestLoggingMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request: Request, call_next):
        request_id = request.headers.get("X-Request-ID", str(uuid4()))

        # structlog contextvars에 request 정보 바인딩
        structlog.contextvars.clear_contextvars()
        structlog.contextvars.bind_contextvars(
            request_id=request_id,
            method=request.method,
            path=request.url.path,
            client_ip=request.client.host,
        )

        start_time = time.perf_counter()

        response = await call_next(request)

        duration_ms = (time.perf_counter() - start_time) * 1000
        logger.info(
            "request_completed",
            status_code=response.status_code,
            duration_ms=round(duration_ms, 2),
        )

        response.headers["X-Request-ID"] = request_id
        response.headers["X-Response-Time"] = f"{duration_ms:.2f}ms"
        return response
```

### 프로덕션 포인트

- `X-Request-ID` 전파: 클라이언트 → API → Celery Worker → 외부 서비스까지 추적
- `contextvars` 사용으로 같은 요청 내 모든 로그에 request_id 자동 포함
- 헬스체크 경로는 로깅에서 제외 (로그 노이즈 방지)
- Body 로깅은 디버그 모드에서만, 크기 제한(1KB) 적용

## 2.3 Rate Limiter (core/rate_limiter.py)

### Token Bucket 알고리즘 (Redis 기반)

```python
class TokenBucketRateLimiter:
    """
    Redis 기반 분산 Token Bucket Rate Limiter.
    여러 API 서버 인스턴스에서 일관된 제한 적용.
    """

    def __init__(self, redis: Redis, rate: float, capacity: int):
        self.redis = redis
        self.rate = rate            # 초당 토큰 보충 속도
        self.capacity = capacity    # 버킷 최대 용량

    async def is_allowed(self, key: str) -> tuple[bool, dict]:
        """Lua 스크립트로 원자적 토큰 소비"""
        # Lua 스크립트: EVAL로 race condition 없이 처리
        # 1. 마지막 리필 이후 경과 시간 계산
        # 2. 토큰 보충 (capacity 초과 안 함)
        # 3. 토큰 1개 소비 시도
        # 4. 남은 토큰 수, 다음 리필 시간 반환
        ...
```

### Rate Limit 미들웨어

```python
class RateLimitMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        # 키 결정: API Key > IP 주소
        key = extract_rate_limit_key(request)
        allowed, info = await rate_limiter.is_allowed(key)

        if not allowed:
            return JSONResponse(
                status_code=429,
                content={"error": {"code": "RATE_LIMIT_EXCEEDED", ...}},
                headers={
                    "Retry-After": str(info["retry_after"]),
                    "X-RateLimit-Limit": str(info["limit"]),
                    "X-RateLimit-Remaining": str(info["remaining"]),
                },
            )

        response = await call_next(request)
        # 응답 헤더에 남은 횟수 포함
        response.headers["X-RateLimit-Remaining"] = str(info["remaining"])
        return response
```

### 프로덕션 포인트

- Lua 스크립트로 원자적 연산 → Redis 분산 환경에서 race condition 없음
- 엔드포인트별 다른 제한: `/chat` (10 req/min), `/documents/ingest` (5 req/min)
- `Retry-After` 헤더로 클라이언트에게 재시도 시점 알림
- Sliding Window Counter도 고려 가능하지만, Token Bucket이 burst 허용 측면에서 유리

## 2.4 서킷 브레이커 (core/circuit_breaker.py)

### 3-State 서킷 브레이커

```
CLOSED (정상)
  → 실패율 >= threshold → OPEN (차단)
      → timeout 경과 → HALF_OPEN (시험)
          → 성공 → CLOSED
          → 실패 → OPEN
```

### 구현

```python
class CircuitBreaker:
    def __init__(
        self,
        failure_threshold: int = 5,      # 연속 실패 N번이면 열림
        recovery_timeout: float = 30.0,   # OPEN 상태 유지 시간(초)
        half_open_max_calls: int = 3,     # HALF_OPEN에서 시험 호출 수
    ):
        self.state = CircuitState.CLOSED
        self.failure_count = 0
        self.last_failure_time: float | None = None
        ...

    async def call(self, func, *args, **kwargs):
        if self.state == CircuitState.OPEN:
            if self._should_try_recovery():
                self.state = CircuitState.HALF_OPEN
            else:
                raise CircuitBreakerOpenError(...)

        try:
            result = await func(*args, **kwargs)
            self._on_success()
            return result
        except Exception as e:
            self._on_failure()
            raise
```

### 프로덕션 포인트

- LLM 서비스별 독립 서킷 브레이커 인스턴스 (vLLM, Ollama, OpenAI 각각)
- 상태 변경 시 로그 + 메트릭 기록 (Prometheus counter)
- Redis에 상태 저장하면 분산 환경에서도 공유 가능 (선택적)
- Fallback 체인: vLLM → Ollama → OpenAI (서킷이 열린 서비스 건너뜀)

## 2.5 FastAPI 의존성 주입 (api/dependencies.py)

### 의존성 팩토리 패턴

```python
# DB 세션
async def get_db() -> AsyncGenerator[AsyncSession, None]:
    async with async_session_factory() as session:
        yield session

# Redis 클라이언트
async def get_redis() -> Redis:
    return RedisManager.get_client()

# Milvus 클라이언트
async def get_milvus() -> MilvusClient:
    return MilvusManager.get_client()

# Settings
def get_settings() -> Settings:
    return settings  # 캐시된 싱글턴

# 리포지토리 (DB 세션 의존)
async def get_document_repo(
    db: AsyncSession = Depends(get_db),
) -> DocumentRepository:
    return DocumentRepository(db)

# RAG Retriever (여러 의존성 조합)
async def get_retriever(
    milvus: MilvusClient = Depends(get_milvus),
    settings: Settings = Depends(get_settings),
) -> Retriever:
    return Retriever(milvus=milvus, settings=settings)
```

### 프로덕션 포인트

- 의존성 체인이 깊어질 수 있으므로, 핵심 리소스(DB, Redis, Milvus)는 `app.state`에서 직접 관리
- 테스트 시 `app.dependency_overrides`로 Mock 주입 가능
- `Annotated[AsyncSession, Depends(get_db)]` 타입 별칭으로 반복 코드 줄임
