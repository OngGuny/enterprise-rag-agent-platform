```python
class Settings(BaseSettings):

        env_file=".env",

    # App

    log_level: str = "INFO"

    # Database
    database_url: str
    db_pool_size: int = 20
    db_max_overflow: int = 10
    db_pool_timeout: int = 30

    # Redis
    redis_url: str = "redis://localhost:6379"
    redis_max_connections: int = 50

    # Milvus
    milvus_host: str = "localhost"
    milvus_port: int = 19530
    milvus_collection_name: str = "documents"

    # LLM
    openai_api_key: str = ""
    ollama_base_url: str = "http://localhost:11434"
    vllm_base_url: str = "http://localhost:8001"
    default_model: str = "gpt-4o-mini"

    # Embedding
    embedding_model: str = "bge-m3"
    embedding_dim: int = 1024

    # Celery
    celery_broker_url: str = "redis://localhost:6379/1"
    celery_result_backend: str = "redis://localhost:6379/2"
```

### 프로덕션 포인트

- `env_nested_delimiter`로 중첩 설정 지원 (예: `DATABASE__POOL_SIZE=20`)
- `@lru_cache`로 Settings 싱글턴 보장: `get_settings()` 함수로 접근
- `.env.example` 파일 제공하되 시크릿은 절대 커밋 안 함
- Validator로 필수 설정 검증 (예: production 모드에서 `openai_api_key` 필수)

## 1.3 FastAPI 앱 진입점 (main.py)

### Lifespan 패턴으로 리소스 관리

```python
@asynccontextmanager
async def lifespan(app: FastAPI):
    # === Startup ===
    # 1. 구조화 로깅 초기화
    # 2. DB 엔진 + 테이블 확인 (Alembic migration 체크)
    # 3. Redis 연결 풀 생성
    # 4. Milvus 연결 + 컬렉션 존재 확인
    # 5. LLM 클라이언트 헬스체크 (Ollama ping 등)
    logger.info("startup_complete", services=["db", "redis", "milvus"])

    yield  # 앱 실행

    # === Shutdown ===
    # 1. Milvus 연결 해제
    # 2. Redis 연결 풀 종료
    # 3. DB 엔진 dispose
    # 4. 진행 중인 Celery 태스크 graceful shutdown 시그널
    logger.info("shutdown_complete")
```

### 프로덕션 포인트

- lifespan 내에서 연결 실패 시 `HealthCheck` 상태를 degraded로 전환 (앱은 떠있되, 해당 기능 비활성)
- `app.state`에 공유 리소스 저장 (Redis pool, Milvus connection 등)
- 미들웨어 순서: CORS → RequestLogging → RateLimiting → PrometheusMetrics

## 1.4 PostgreSQL + SQLAlchemy Async 세션

### 엔진 & 세션 팩토리

```python
engine = create_async_engine(
    settings.database_url,
    pool_size=settings.db_pool_size,
    max_overflow=settings.db_max_overflow,
    pool_timeout=settings.db_pool_timeout,
    pool_pre_ping=True,          # 연결 유효성 사전 검사
    pool_recycle=1800,           # 30분마다 연결 재생성
    echo=settings.debug,
)

async_session_factory = async_sessionmaker(
    engine,
    class_=AsyncSession,
    expire_on_commit=False,      # 커밋 후 객체 재사용 가능
)
```

### DB 세션 DI (FastAPI Depends)

```python
async def get_db_session() -> AsyncGenerator[AsyncSession, None]:
    async with async_session_factory() as session:
        try:
            yield session
            await session.commit()
        except Exception:
            await session.rollback()
            raise
```

### ORM 모델 베이스

- `Base = DeclarativeBase` with common mixin: `id` (UUID), `created_at`, `updated_at`
- `TimestampMixin`으로 created_at/updated_at 자동 관리
- Alembic async migration 설정 (env.py에서 `run_async_migrations`)

### 프로덕션 포인트

- `pool_pre_ping=True`: 네트워크 단절 후 stale connection 방지
- `expire_on_commit=False`: async 환경에서 lazy loading 문제 방지
- Connection Pool 모니터링: pool.status() 메트릭을 Prometheus로 노출

## 1.5 Redis 연결 관리

### 비동기 Redis 클라이언트

```python
from redis.asyncio import ConnectionPool, Redis

class RedisManager:
    _pool: ConnectionPool | None = None

    @classmethod
    async def connect(cls, url: str, max_connections: int = 50):
        cls._pool = ConnectionPool.from_url(
            url,
            max_connections=max_connections,
            decode_responses=True,      # 자동 문자열 디코딩
            socket_timeout=5.0,
            socket_connect_timeout=5.0,
            retry_on_timeout=True,
        )

    @classmethod
    def get_client(cls) -> Redis:
        if cls._pool is None:
            raise RuntimeError("Redis not connected")
        return Redis(connection_pool=cls._pool)

    @classmethod
    async def disconnect(cls):
        if cls._pool:
            await cls._pool.disconnect()
```

### 용도별 DB 분리

- DB 0: 시맨틱 캐시 (LLM 응답)
- DB 1: Celery broker
- DB 2: Celery result backend
- DB 3: Rate Limiter 카운터
- DB 4: 세션/대화 히스토리 (선택)

## 1.6 Milvus 연결 관리

### PyMilvus 비동기 래퍼

```python
from pymilvus import MilvusClient

class MilvusManager:
    _client: MilvusClient | None = None

    @classmethod
    async def connect(cls, host: str, port: int):
        cls._client = MilvusClient(uri=f"http://{host}:{port}")
        # 연결 확인
        cls._client.list_collections()
        logger.info("milvus_connected", host=host, port=port)

    @classmethod
    def get_client(cls) -> MilvusClient:
        if cls._client is None:
            raise RuntimeError("Milvus not connected")
        return cls._client
```

### 프로덕션 포인트

- PyMilvus 2.4+의 MilvusClient 사용 (기존 connections API 대비 단순화)
- 연결 타임아웃 설정
- 재연결 로직 (lifespan에서 retry with backoff)

## 1.7 헬스체크 엔드포인트

### 다층 헬스체크

```python
@router.get("/health")
async def health_check():
    return {"status": "ok"}  # Kubernetes liveness probe용

@router.get("/health/ready")
async def readiness_check():
    checks = {
        "database": await check_db(),
        "redis": await check_redis(),
        "milvus": await check_milvus(),
    }
    all_healthy = all(v["status"] == "ok" for v in checks.values())
    status_code = 200 if all_healthy else 503
    return JSONResponse(
        content={"status": "ready" if all_healthy else "degraded", "checks": checks},
        status_code=status_code,
    )
```

## 1.8 구조화 로깅 (structlog)

### 설정

```python
import structlog

def setup_logging(log_level: str = "INFO"):
    structlog.configure(
        processors=[
            structlog.contextvars.merge_contextvars,
            structlog.processors.add_log_level,
            structlog.processors.TimeStamper(fmt="iso"),
            structlog.processors.StackInfoRenderer(),
            structlog.processors.format_exc_info,
            structlog.processors.JSONRenderer(),  # 프로덕션: JSON
        ],
        wrapper_class=structlog.make_filtering_bound_logger(
            getattr(logging, log_level)
        ),
        context_class=dict,
        logger_factory=structlog.PrintLoggerFactory(),
    )
```

### 프로덕션 포인트

- JSON 포맷 → ELK/Loki에서 파싱 가능
- `contextvars`로 request_id, user_id를 모든 로그에 자동 주입
- 개발 환경에서는 `ConsoleRenderer`로 읽기 쉬운 출력
